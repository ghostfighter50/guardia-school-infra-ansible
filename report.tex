\section{Résumé Exécutif}

Ce document constitue la documentation technique finale du dépôt Ansible actuellement versionné pour le projet Guardia School. Il décrit l'état réel du code, les composants effectivement automatisés et les intégrations documentées autour du dépôt.

Le périmètre implémenté est centré sur Linux : création du compte de service Ansible, déploiement d'un Vault local, durcissement SSH avec TOTP, configuration UFW, agent SNMP et provisioning d'un utilisateur géré. Les intégrations Centreon et OPNsense sont documentées, mais ne sont pas déployées par les playbooks principaux.

  \item \textbf{7 rôles} réellement utilisés : \texttt{service\_account}, \texttt{vault\_server}, \texttt{common}, \texttt{ssh\_hardening}, \texttt{ufw\_firewall}, \texttt{snmp\_agent}, \texttt{managed\_user}
  \item Vault local en \textbf{mode labo} : stockage fichier, HTTP, KV v2, unseal manuel après redémarrage
  \item SSH durci sur le port \texttt{2222} avec \textbf{TOTP via PAM} pour les comptes humains et exemption du compte de service Ansible
  \item SNMP en \textbf{v2c read-only} avec restriction par IP source, cohérent avec le template \texttt{snmpd.conf.j2}
  \item Centreon intégré via \textbf{SNMP et configuration UI}, avec contrôles explicites pour sortir l'hôte de l'état \texttt{PENDING}
  \item OPNsense documenté dans un guide séparé, sans rôle Ansible dédié dans l'état actuel du dépôt
\end{itemize}

\section{Introduction et Objectifs}

\subsection{Contexte}

Le dépôt vise à rendre reproductible le durcissement initial d'une infrastructure Linux à partir d'un contrôleur Ansible. L'approche retenue repose sur des playbooks simples, idempotents, lisibles et focalisés sur les besoins réellement présents dans le code.

\subsection{Objectifs techniques effectivement couverts}

\begin{itemize}
  \item Préparer un compte de service Ansible utilisable en SSH par clé
  \item Déployer HashiCorp Vault localement sur le contrôleur
  \item Durcir SSH avec bannière légale, port non standard et 2FA TOTP
  \item Appliquer une politique UFW de type \texttt{deny-all inbound}
  \item Déployer SNMP sur les cibles Linux pour une supervision externe
  \item Proviser un utilisateur géré avec mot de passe et TOTP stockés dans Vault
  \item Documenter l'intégration manuelle avec Centreon et OPNsense
\end{itemize}

\subsection{Hors périmètre actuel du dépôt}

\begin{itemize}
  \item Automatisation Windows ou WinRM
  \item Agent Centreon déployé par Ansible
  \item Rotation automatique planifiée des secrets
  \item Inventaire automatique généré par le flux principal des playbooks
  \item Configuration automatisée d'OPNsense
\end{itemize}

  \item Bootstrap initial : SSH sur le port \texttt{22}
  \item Après durcissement : SSH sur le port \texttt{2222}

\section{Architecture Ansible}

\subsection{Principe de fonctionnement}

Ansible se connecte aux cibles Linux via SSH, exécute les tâches YAML, puis délègue le stockage des secrets à Vault via son API HTTP. Aucun agent n'est installé sur les cibles Linux.

\section{Composants de Sécurité}

\subsection{SSH et 2FA}

Le fichier \texttt{roles/ssh\_hardening/templates/99-hardening.conf.j2} configure :

\begin{itemize}
  \item le port \texttt{2222}
  \item \texttt{PermitRootLogin no}
  \item \texttt{AuthenticationMethods publickey,keyboard-interactive}
  \item une exception \texttt{Match User} pour le compte de service Ansible en clé seule
\end{itemize}

Les secrets TOTP sont générés par Ansible puis stockés dans Vault. Les administrateurs n'exécutent pas \texttt{google-authenticator} manuellement dans le flux nominal ; ils récupèrent le secret depuis Vault pour l'enrôler dans une application d'authentification.

\subsection{UFW}

Le rôle \texttt{ufw\_firewall} applique une politique \texttt{deny-all inbound}. Les exceptions documentées sont :

\begin{itemize}
  \item SSH sur \texttt{TCP/2222}
  \item SNMP sur \texttt{UDP/161} depuis \texttt{snmp\_allowed\_source}
\end{itemize}

\subsection{SNMP}

Le rôle \texttt{snmp\_agent} déploie \texttt{snmpd} en \textbf{SNMPv2c} avec communauté en lecture seule et restriction par IP source. Le dépôt ne configure ni SNMPv3, ni authentification SHA, ni chiffrement AES.

\end{tcolorbox}

\subsection{HashiCorp Vault}

Vault est déployé en single-node avec stockage fichier. Les caractéristiques observables dans le code sont les suivantes :

\begin{itemize}
  \item écoute sur \texttt{0.0.0.0:8200}
  \item fichier de configuration \texttt{/etc/vault.d/vault.hcl}
  \item données dans \texttt{/opt/vault/data}
  \item initialisation Shamir \texttt{5/3}
  \item stockage des clés d'unseal dans \texttt{/root/vault\_init.json}
  \item moteur KV v2 monté sur \texttt{secret/}
  \item secrets seedés : \texttt{infra/bootstrap} et \texttt{infra/admin}
\end{itemize}

Le dépôt ne configure pas aujourd'hui l'auto-unseal, la rotation automatique des secrets, ni TLS sur Vault.

\section{Supervision Centreon et OPNsense}

\subsection{Centreon}

L'intégration Centreon est documentée via \texttt{docs/setup-centreon.md}. Le flux réel repose sur :

\begin{itemize}
  \item le plugin pack Linux SNMP
  \item la création manuelle des hôtes dans l'UI Centreon
  \item l'export de configuration vers le poller
  \item la vérification des services \texttt{gorgoned}, \texttt{cbd} et \texttt{centengine}
  \item le reschedule manuel des checks pour sortir les hôtes et services de l'état \texttt{PENDING}
\end{itemize}

Le dépôt ne déploie pas d'agent Centreon sur les cibles.

\subsection{OPNsense}

OPNsense est présent dans l'inventaire comme référence de passerelle ou de pare-feu. La configuration est décrite dans \texttt{docs/setup-opnsense.md} et reste \textbf{manuelle}. Le flux principal Ansible n'applique aucune règle OPNsense.

\section{Déploiement sur Machine Vierge}

\subsection{Pré-requis}

\begin{itemize}
  \item Contrôleur Linux avec Git, Ansible et accès réseau aux cibles
  \item Accès SSH + sudo à la cible Linux initiale
  \item Clé ED25519 générée par \texttt{bootstrap.sh} ou manuellement
  \item Inventaire statique mis à jour dans \texttt{inventory/hosts.yml}
  \item IP du poller définie dans \texttt{inventory/group\_vars/all.yml}
\end{itemize}

\section{Conclusion}

Le dépôt fournit aujourd'hui une base propre et cohérente pour automatiser le durcissement initial d'une infrastructure Linux avec Vault, SSH/TOTP, UFW et SNMP. Les parties Centreon et OPNsense sont présentes dans la documentation et dans l'exploitation, mais restent volontairement hors du flux Ansible principal.

Cette version du rapport sert de référence fidèle au code réellement versionné. Toute évolution future du dépôt devra conserver cette discipline : documenter uniquement ce qui existe dans les playbooks, rôles, variables et guides effectivement présents.
